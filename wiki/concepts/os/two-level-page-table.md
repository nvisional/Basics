# 二级页表 — 地址转换全过程

## 直觉

页表不是特殊硬件，就是放在物理内存里的"数组"。MMU 做地址转换本质上就是**算下标 → 读内存 → 剥出页框号 → 拼地址**，重复两遍而已。

整个过程的核心反复做一件事：**基址 + 索引 × 条目大小 = 目标地址**。

只有 CR3 存的是真正的字节级物理地址，PDE 和 PTE 里存的都是**页框号（以 4KB 为单位的偏移量）**，末尾低 12 位 MMU 自动补零。

---

## 虚拟地址的字段划分

32 位虚拟地址被硬件按位切分为三段（10-10-12）：

```
┌─────────────────────┬─────────────────────┬──────────────────────────┐
│   页目录号 PDI       │   页表索引 PTI        │    页内偏移 Offset         │
│   bit[31:22] 10位    │   bit[21:12] 10位     │    bit[11:0] 12位         │
└─────────────────────┴─────────────────────┴──────────────────────────┘
```

硬件通过**位运算**提取各字段（组合逻辑电路直接完成，不走软件指令）：

```
PDI    = (VA >> 22) & 0x3FF       // 取高 10 位
PTI    = (VA >> 12) & 0x3FF       // 取中 10 位
Offset =  VA        & 0xFFF       // 取低 12 位
```

换个角度看，整个过程只用到了**两次偏移**：

| 次数 | 偏移量 | 作用 |
|------|--------|------|
| 第一次 | 高 20 位（PDI + PTI，即虚拟页号 VPN） | 经过两级间接查表，找到物理页框号 |
| 第二次 | 低 12 位（Offset） | 在物理页框内定位最终字节 |

页表之所以分两级，是因为一张能映射全部 4GB 空间的大页表（2^20 项 × 4 字节 = 4MB）太浪费——多数进程用不满，但页表本身必须连续存放。拆成两级后，每级恰好一页（4KB），可以按需分配。

---

## Step 1: 从 CR3 定位页目录项 PDE

**CR3** 是 x86 的控制寄存器（PDBR，Page Directory Base Register），存着页目录表在物理内存中的起始物理地址（**字节级，真正的完整地址**）。

```
PDE 物理地址 = CR3 + PDI × 4
```

每个 PDE 占 4 字节，PDI × 4 就是字节偏移。**这是一次物理内存读取。**

### PDE 的结构

```
PDE (32 bits):
┌──────────────────────────────────────┬──┬──┬──┬──┬──┬──┬──┬──┐
│    页表的物理页框号 (bit 31:12)         │  │  │  │  │  │P │  │
│    只有高 20 位，低 12 位不存在          │  │  │  │  │  │  │  │
└──────────────────────────────────────┴──┴──┴──┴──┴──┴──┴──┴──┘
                                              P (Present)：页表是否在内存中
```

PDE 里存的**不是**完整的 32 位物理地址，而是以 4KB 为单位的**页框号**。MMU 用的时候左移补 0：

```
页表物理基址 = PDE[31:12] << 12      // 末尾自动补 12 个 0
```

P 位 = 0 时触发**缺页异常（Page Fault）**，由操作系统处理。

---

## Step 2: 从页表定位页表项 PTE

```
PTE 物理地址 = 页表基址 + PTI × 4
```

**这又是一次物理内存读取。** 每个 PTE 同样占 4 字节。

### PTE 的结构

```
PTE (32 bits):
┌──────────────────────────────────────┬──┬──┬──┬──┬──┬──┬──┬──┐
│    目标物理页框号 (bit 31:12)           │  │  │  │  │  │P │  │
└──────────────────────────────────────┴──┴──┴──┴──┴──┴──┴──┴──┘

最终物理页基址 = PTE[31:12] << 12
```

同样检查 P 位，为 0 则缺页。

---

## Step 3: 拼出最终物理地址

```
最终物理地址 = 物理页基址 + Offset
```

---

## 完整链路（全在物理内存里）

```
VA:  [ PDI ][ PTI ][ Offset ]

(1) 读 PDE:  去 addr = CR3 + PDI×4     → 读 4 字节 → 得页表物理页框号
(2) 读 PTE:  去 addr = 页表基址 + PTI×4  → 读 4 字节 → 得最终物理页框号
(3) 读数据:  去 addr = 物理页基址 + Offset → 读/写真正的数据
```

**一次数据访问总共要访问 3 次物理内存**（2 次查表 + 1 次取数据）。

---

## 每层存的"偏移量"单位不同

全部串联起来看：

```
CR3          → 真正的字节级物理地址            (单位: 字节)
    +
PDI × 4      → 页目录内的第几项               (单位: 4 字节 = 一个 PDE)
    ↓
PDE[31:12]   → 页表的物理页框号               (单位: 4KB = 一页)
    +
PTI × 4      → 页表内的第几项                 (单位: 4 字节 = 一个 PTE)
    ↓
PTE[31:12]   → 目标页的物理页框号             (单位: 4KB = 一页)
    +
Offset       → 页内字节偏移                   (单位: 1 字节)
```

**只有 CR3 存的是完整字节地址，PDE 和 PTE 里存的都是页框号——本质上就是"第几个 4KB 物理页"。** 这也是为什么 4KB 页的物理地址低 12 位总是 0——PDE/PTE 里压根没存，也不需要存，反正是 4KB 对齐的。

---

## 为什么要有 TLB

既然每次地址转换都要多读 2 次内存（多级页表就更离谱了），开销巨大。所以 CPU 内部有一块高速缓存叫 **TLB（Translation Lookaside Buffer，快表）**，把常用的 PDE/PTE 缓存起来。TLB 命中时直接跳过那两次内存读，只需一次就能完成地址转换。

---

## 为什么是 10-10-12 划分

| 字段 | 位数 | 可索引条目 | 条目大小 | 表大小 |
|------|------|-----------|---------|--------|
| PDI  | 10   | 1024 PDE  | 4 字节  | 4KB（恰好 1 页） |
| PTI  | 10   | 1024 PTE  | 4 字节  | 4KB（恰好 1 页） |
| Offset | 12 | 4096 字节 | —      | —        |

每级页表刚好一页，不会跨页，OS 管理起来非常方便。这是 x86 经典分页的设计动机。

---

## 设计权衡

### 为什么分两级而不是一级？

| 方案 | 优点 | 缺点 |
|------|------|------|
| 一级大页表（2^20 项 × 4B = 4MB） | 查表只需一次内存访问 | 每个进程都要 4MB 连续物理内存；大部分页表项空闲 |
| 二级页表（1024 PDE + 按需 PTE） | 按需分配，未用到的页表不占内存 | 多一次内存访问（3 次 vs 2 次） |

**结论**：空间换时间的经典反面——用多一次内存访问换极大的内存节省，再加上 TLB 缓存让大多数情况下感受不到这次额外开销。

### 为什么是 4KB 页？

| 页面大小 | 优点 | 缺点 |
|---------|------|------|
| 小页（4KB） | 内部碎片少，精细控制 | 页表项多，查表开销大 |
| 大页（2MB / 1GB） | 页表项少，TLB 覆盖率高 | 内部碎片严重，不够灵活 |

x86 选择 4KB 作为基础页大小，同时支持大页（PSE, 2MB / 1GB）作为补充。现代 OS 对热点区域使用大页（transparent huge pages），冷数据用小页，两者结合。

### 为什么 PDE 和 PTE 只存高 20 位？

页表本身必须 4KB 对齐——如果允许放在任意字节地址，页表可能跨页，管理复杂度暴增。强制 4KB 对齐后，物理地址低 12 位永远是 0，所以 PDE/PTE 里直接省略，省下 12 位给控制标志（Present、R/W、U/S、Dirty、Accessed 等），一个 32 位字段同时承载地址 + 权限控制，设计精巧。

## 虚拟地址与物理地址的先后关系

### 程序只发虚拟地址

CPU 执行的每一刻，指令指针和访存地址**全部是虚拟地址**。程序从头到尾不知道物理地址长什么样：

```
程序的视角:            MMU 的视角:

  mov eax, [0x4000]  ──→ VA=0x4000
                           │
                           ├→ 查页表 (PDI → PDE → PTI → PTE)
                           │
                           └→ PA=0x01FFF000  ← 物理地址是翻译结果
```

虚拟地址在前，物理地址是**查表结果**。程序不知道、也不需要知道物理地址。

### 页表遍历用的是物理地址

但 MMU 查页表的过程中，读 PDE、读 PTE 用的都是物理地址——这是唯一"物理地址跑在虚拟地址前面"的时刻，但对程序完全透明。

### 分工

| 角色 | 做什么 |
|------|--------|
| **OS** | 管理物理内存，决定给每个进程分配哪些页框，把映射关系填入页表 |
| **MMU** | 硬件查表工，OS 填好页表后，MMU 自动做 VA→PA 翻译 |

OS 是决策者，MMU 是执行者。MMU 不管分配，只管照着页表机械翻译。

---

## 页表即约束

### 页表不指向别处，所以物理地址天然限制在分配的页框内

OS 给进程分配了 N 个物理页框后，页表里 PTE 指向的物理页框号就只能是这 N 个之一。**不是靠"检查"或"拦截"，而是页表里压根就没有指向第 N+1 个页框的条目。**

```
进程的物理地址落点 = { PTE[0..N-1] 里记录的页框号 }

页表里没填的 → P 位 = 0 → 缺页异常 → OS 介入
页表里填了的 → P 位 = 1 → 物理页框号必在分配的 N 个之内
```

打个比方：进程是按地图走路的人，页表是地图，OS 画地图。OS 压根不在地图上画出第 N+1 个目的地，所以进程天然走不到别处。

### OS 建立映射时的约束

OS 在填 PTE（建立 VA→PA 映射）时，只能从分配给该进程的页框池里选：

```
缺页异常 → OS 介入
              │
              ├── 固定分配: 从进程的 N 个页框中挑一个踢掉，装入新页
              │
              └── 可变分配: 可以从全局池分配新页框，也可能回收其他进程的页框
```
