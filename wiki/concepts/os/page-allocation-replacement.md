# 页框分配策略与页面置换范围

## 直觉

OS 给进程分配物理页框时，有两个独立的决策维度：

- **分配多少**：给进程几个页框？这个数量在运行中能变吗？
- **踢谁的**：缺页时从谁的页框池里淘汰页面？

两个维度正交组合，得到三种有意义的策略。

---

## 两个正交维度

```
                  局部置换                      全局置换
              (只踢自己的页)                 (从所有进程的池子里抢)

固定分配    ✅ 固定 + 局部                    ❌ 不存在
(页框数不变)  每进程固定N个                   全局意味着页框可跨进程流动
             缺页只置换自己的                 与"固定"矛盾

可变分配    ✅ 可变 + 局部                    ✅ 可变 + 全局
(页框数动态)  页框数可调                      缺页时全局 LRU 链表上抢
             但置换范围仍限于自己              活跃进程自然获得更多页框
```

---

## 固定分配 + 局部置换

- **"固定"**：OS 在进程创建时分配 N 个物理页框，到进程结束都不变
- **"局部"**：进程缺页时，只能从自己这 N 个页框里选一页踢出去

| 优点 | 缺点 |
|------|------|
| 简单公平，一个进程的频繁缺页不影响别人 | N 设小了频繁缺页，设大了浪费内存 |
| 行为可预测 | 不能根据实际运行行为动态调整 |

---

## 可变分配 + 局部置换

- OS 根据缺页频率（PFF, Page Fault Frequency）动态调整页框数
- 频繁缺页 → 多给几个页框；很少缺页 → 收回一些
- 置换范围仍然只限于本进程

核心思想：**用缺页频率作为反馈信号**，自适应调整每个进程的工作集大小。

---

## 可变分配 + 全局置换（Linux 实际使用）

- 所有进程共享全局空闲页框池
- 缺页时从全局 LRU 链表上淘汰最久未用的页，不关心这页属于谁
- 活跃进程自然获得更多页框，不活跃的慢慢被挤出去
- 通过 memory cgroup、rlimit 提供 per-group 限制作为补充

| 优点 | 缺点 |
|------|------|
| 内存利用率高，自适应负载变化 | 一个进程可能被另一个拖垮 |
| 没有人为的固定配额 | 需要更复杂的回收和记账机制 |

---

## 设计权衡

### 为什么不能固定 + 全局？

全局置换意味着页框可以在进程之间流动，一个进程缺页可能把另一个进程的页踢出去。但如果分配又是"固定"的——每个进程的页框数不变——这两个要求互相矛盾：全局置换必然改变各进程持有的页框数。所以这一格在逻辑上不成立。

### 局部 vs 全局的取舍

| 局部置换 | 全局置换 |
|---------|---------|
| 进程间隔离好 | 内存利用率更高 |
| 工作集估计不准时浪费内存 | 可能出现"扰邻"（一个坏进程拖慢全局） |
| 实现简单 | 需要全局 LRU 等更复杂的数据结构 |

现代 OS（Linux、Windows）基本走全局路线，因为实际工作负载中局部置换的浪费太大。
